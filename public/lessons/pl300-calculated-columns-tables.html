Calculated Columns vs Calculated Tables

Calculated columns add derived fields to existing tables (row-by-row calculations). Calculated tables create entirely new tables using DAX. Both add flexibility but impact model size and refresh performance.

Key Principle
Use calculated columns for row-level derivations (concatenation, categorization, flags). Use calculated tables for lookups, distinct lists, or aggregations. Prefer Power Query when possible for better performance.

Calculated Columns

✓ When to Use Calculated Columns▼

Use Cases:

✓ Concatenate fields (FirstName + LastName)
✓ Categorize values (Age → AgeGroup)
✓ Extract components (Month from Date)
✓ Calculate ratios (Profit/Revenue)
✓ Boolean flags (IsExpired, IsHighValue)
✓ Format values (currency, percentages)

Examples:

FullName = [FirstName] & " " & [LastName] AgeGroup = IF([Age]<18,"Under 18", IF([Age]<65,"Adult","Senior")) Profit = [Sales] - [Cost] IsWeekend = [DayOfWeek]="Saturday" || [DayOfWeek]="Sunday"
⚠️ When NOT to Use Calculated Columns▼
⚠ Aggregations (use measures instead)
⚠ Complex multi-table lookups (use relationships)
⚠ Time intelligence (use measures + date table)
⚠ Context-dependent values (measures better)
⚠ Simple lookups (RELATED() in measures)

Performance Cost:

⚠ Each row calculated at refresh
⚠ Increases file size per column
⚠ Slower refresh (M-side formulas better)

Calculated Tables

✓ When to Use Calculated Tables▼

Use Cases:

✓ Distinct lists (unique products, categories)
✓ Lookup tables (missing dimension data)
✓ Top-N tables (Top 10 products by sales)
✓ Aggregated data (summary tables)
✓ Role-playing dimensions (duplicate dates)
✓ Parameterized tables (dynamic lists)

Examples:

TopProducts = TOPN(10, FactSales, [Total Sales], DESC) Categories = DISTINCT(FactSales[Category]) DateCalendar = CALENDAR( DATE(2020,1,1), DATE(2025,12,31))
⚠️ When NOT to Use Calculated Tables▼
⚠ For simple transformations (Power Query better)
⚠ Large datasets (performance hit)
⚠ When Power Query exists (use source instead)
⚠ Static data (load from source)

Performance Cost:

⚠ Full table refresh each time
⚠ No incremental loading
⚠ Increases model size significantly

Calculated Columns vs Tables

⚖️ When to Use Each▼

Use Calculated Columns When:

✓ Row-level formula needed
✓ Based on existing table data
✓ Used in filters or slicers
✓ Small overhead per row

Use Calculated Tables When:

✓ Need new table structure
✓ Table not in source data
✓ Dynamic lookup lists
✓ Dimension supplement

Use Power Query Instead When:

✓ Performance critical (large datasets)
✓ Complex transformations
✓ Can modify source data
✓ Incremental refresh needed

Real-World Scenarios

Scenario 1: Calculated Column - Full Name
Concatenate first and last names

Formula:

FullName = [FirstName] & " " & [LastName]

Why Column:

✓ Row-level formula
✓ Based on existing columns
✓ Needed for slicers/filters

File Impact:

✓ Minimal (text concatenation)
Scenario 2: Calculated Column - Age Category
Categorize customers by age group

Formula:

AgeCategory = IF([Age]<18,"Teen", IF([Age]<35,"Young Adult", IF([Age]<55,"Adult","Senior")))

Why Column:

✓ Per-customer categorization
✓ Used in pivot tables, slicers

Alternative:

⚠ Could do in Power Query (better)
Scenario 3: Calculated Table - Top 10 Products
Dynamic list of best-selling products

Formula:

Top10Products = TOPN(10, VALUES(FactSales[ProductID]), [Total Sales], DESC)

Why Table:

✓ Creates new table
✓ Dynamic (updates with sales)
✓ Used for relationship

Use Case:

✓ Filters report to top products
Scenario 4: Calculated Table - Date Calendar
Generate complete date dimension

Formula:

DimDate = CALENDAR( DATE(2020,1,1), DATE(2025,12,31))

Why Table:

✓ Complete date coverage
✓ No gaps in date range
✓ Can add columns for fiscal year

Best Practices

✓ Calculated Column Best Practices
▼
✓ Keep formulas simple and readable
✓ Use descriptive names (not CalcCol1)
✓ Prefer Power Query for large transforms
✓ Test on small dataset first
✓ Monitor file size growth
✓ Use measures for aggregations
✓ Document complex formulas
✓ Calculated Table Best Practices▼
✓ Use for dimension supplements only
✓ Avoid for large fact tables
✓ Consider Power Query alternative
✓ Name tables clearly
✓ Mark as hidden if not for users
✓ Create relationships to fact tables
✓ Test performance impact
⚠️ Common Mistakes▼
⚠ Using calc column for aggregation
⚠ Calc table for transformations (M better)
⚠ Ignoring file size impact
⚠ Complex formulas in columns (hard to maintain)
⚠ Calc tables for large datasets
⚠ Not documenting purpose
