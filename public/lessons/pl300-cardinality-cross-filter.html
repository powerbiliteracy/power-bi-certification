Relationship Cardinality & Cross-Filter Direction

Cardinality defines the relationship type (one-to-many, many-to-many). Cross-filter direction controls how filters propagate. Both critically impact calculations, performance, and user experience.

Key Principle
Cardinality determines relationship type. Cross-filter direction controls how filters flow through relationships. Both affect model behavior, calculations, and performance. Defaults (one-to-many, single direction) work 80% of the time.

Relationship Cardinality Types

1️⃣ One-to-Many (1:*)▼

One row in dimension relates to many rows in fact table.

Example:

DimProduct: FactSales: ProductID (PK) SalesID Name ProductID (FK) Category Amount 1 Product → Many Sales

Characteristics:

✓ Primary key on ONE side
✓ Foreign key on MANY side
✓ Most common relationship type
✓ Standard for star schema

When to Use:

✓ Dimension to Fact
✓ Lookup tables
✓ Parent-child hierarchies
∞ Many-to-Many (*:*)▼

Many rows in one table relate to many rows in another.

Example:

Students: Classes: StudentID (PK) ClassID (PK) Name Name Many Students → Many Classes (via bridge table)

Characteristics:

⚠ MANY on both sides
⚠ Requires bridge table
⚠ Complex calculations
⚠ Performance overhead

When to Use:

⚠ Students ↔ Classes
⚠ Movies ↔ Actors
⚠ Only when truly many-to-many

Best Practice:

Use bridge table instead when possible

Cross-Filter Direction

→ Single Direction (Default)▼

Filters flow ONE way: from ONE side to MANY side.

Example:

DimProduct → FactSales (Dimension) (Fact) Filter Product → Filters Sales Filter Sales ↛ Does NOT filter Product

Behavior:

✓ Select product → sales filtered
✓ Select sales → product NOT filtered
✓ Standard for star schema
✓ Best performance

Use When:

✓ Standard dimension-to-fact
✓ Always filtering fact from dimension
↔ Both Directions (Bidirectional)▼

Filters flow BOTH ways between related tables.

Example:

Table A ↔ Table B (Mutual filtering)

Behavior:

⚠ Select in Table A → Table B filtered
⚠ Select in Table B → Table A filtered
⚠ Slower performance
⚠ Complex logic

Use When:

⚠ Need bidirectional filtering
⚠ Non-standard relationships
⚠ Rare (avoid if possible)
🔒 Both (Limited)▼

Bidirectional with performance optimization (Power BI only).

Characteristics:

✓ Bidirectional filtering
✓ Better performance than "Both"
✓ Optimized for large models
⚠ Power BI Desktop only

Real-World Scenarios

Scenario 1: Standard Star Schema (One-to-Many, Single)
Typical retail data model

Structure:

DimProduct (1) → FactSales (Many)
DimCustomer (1) → FactSales (Many)
DimDate (1) → FactSales (Many)

Cardinality:

✓ All One-to-Many

Cross-Filter:

✓ All Single Direction

Result:

✓ Simple, fast, intuitive
Scenario 2: Many-to-Many via Bridge Table
Students taking multiple classes

Tables:

DimStudent (1) → BridgeStudentClass (Many)
DimClass (1) → BridgeStudentClass (Many)

Cardinality:

⚠ Two One-to-Many relationships
⚠ Bridge table in middle

Result:

✓ Avoids true M:M complexity
✓ Better performance
Scenario 3: Bidirectional (When Needed)
When filters should flow both ways

Use Case:

Table A ↔ Table B (mutual context)

When to Use:

⚠ Non-standard schema
⚠ Peer-to-peer relationships

Cost:

⚠ 10-20% slower performance
⚠ More complex DAX

Best Practices

✓ Cardinality Best Practices▼
✓ Default to One-to-Many (95% of cases)
✓ Dimension PK → Fact FK always
✓ Avoid true Many-to-Many
✓ Use bridge tables instead of M:M
✓ Verify cardinality before creating
✓ Cross-Filter Best Practices▼
✓ Default to Single Direction
✓ Only use Both when necessary
✓ Document why bidirectional needed
✓ Test performance impact
✓ Use CROSSFILTER() in DAX if needed
⚠️ Common Mistakes▼
⚠ Using Many-to-Many instead of bridge
⚠ Setting bidirectional unnecessarily
⚠ Not verifying cardinality
⚠ Wrong side of relationship (FK on ONE)
⚠ Too many bidirectional relationships
