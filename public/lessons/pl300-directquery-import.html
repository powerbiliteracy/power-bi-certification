Choose Between DirectLake, DirectQuery, and Import

PL-300: Pick the right storage mode for performance, freshness, and scale

📦
Import
Data is loaded into the model (VertiPaq).
🔌
DirectQuery
Queries are sent live to the source on every interaction.
🌊
DirectLake
Reads Delta/Parquet files directly from OneLake — no import, no DAX-to-SQL translation.
💡
What changed in the latest exam
DirectLake is now the third storage mode you must compare alongside Import and DirectQuery. It's exclusive to Microsoft Fabric semantic models built on OneLake (Lakehouse / Warehouse). It targets near-Import performance with near-DirectQuery freshness.

📦 Import — when it shines▼
✓ Best raw query performance for visuals
✓ Full DAX, calculated columns, and calculation groups
✓ Works with any data source connector
✗ Stale until next refresh; capacity memory limits dataset size
🔌 DirectQuery — when it shines▼
✓ Always-live data without scheduled refresh
✓ Bypasses model size limits — query the source
✗ Visuals are only as fast as the source can answer DAX-to-SQL
✗ Some DAX (time intelligence, complex measures) is restricted
🌊 DirectLake — when it shines▼
✓ Import-like speed without copying data into the model
✓ Auto-updates as Delta tables are written in the Lakehouse
✓ Full DAX, with automatic fallback to DirectQuery for edge cases
✗ Requires Fabric capacity and a Lakehouse / Warehouse source
✗ No Power Query transformations in the semantic model — shape data upstream

Quick decision guide

🧭
Start here
Ask three questions: How fresh must the data be? How big is it? Is the source already in OneLake (Fabric)?

🚦 Decision flow (text)▼
1. Is the source a Fabric Lakehouse/Warehouse? → DirectLake by default.
2. Otherwise, does the data fit in memory and is daily refresh OK? → Import.
3. Need real-time and source can handle interactive load? → DirectQuery.
4. Mix of needs? → Composite with Dual storage on dimensions.

Composite models, Dual storage & DirectLake fallback

🧩 Composite models▼
Mix Import and DirectQuery sources in one model.
Common pattern: huge fact in DirectQuery, small dimensions in Import.
Watch limited relationships and security boundaries between sources.
🪞 Dual storage mode▼
Set on dimension tables: behaves as Import when joined to Import facts, DirectQuery when joined to DQ facts.
Reduces slow cross-source joins in composite models.
♻️ DirectLake fallback▼
If a query exceeds DirectLake guardrails (row groups, column count, SKU limits), it silently falls back to DirectQuery on SQL endpoint.
Good: queries still return. Bad: performance drops — monitor with Performance Analyzer.
Reduce risk: keep tables tidy (V-Order optimised Delta), partition large facts, use higher Fabric SKUs.

Real-world scenarios

Scenario 1 — 50M-row sales report refreshed nightly from Azure SQL
Choose: Import. The dataset compresses well, freshness target is daily, and Import gives the fastest user experience.
Scenario 2 — Operational dashboard on the warehouse floor, must reflect every minute
Choose: DirectQuery on a tuned source (or DirectLake if running on Fabric Lakehouse). Pair with automatic page refresh.
Scenario 3 — Enterprise has migrated to Fabric, 800M-row Lakehouse fact
Choose: DirectLake. Skip the import refresh, get Import-like performance, stay in sync with the Lakehouse.
Scenario 4 — Marketing wants Snowflake live for ads, but small product dim should be fast
Choose: Composite — DirectQuery on the ad spend fact, Import (or Dual) on the product dimension.

Best practices

📦 Import▼
✓ Use incremental refresh on large facts.
✓ Remove unused columns to shrink the model.
✓ Star schema beats wide flat tables.
🔌 DirectQuery▼
✓ Push aggregates and indexes down to the source.
✓ Use aggregations tables to accelerate common visuals.
✓ Avoid complex DAX measures that translate to expensive SQL.
🌊 DirectLake▼
✓ Curate the Lakehouse with V-Order optimised Delta tables.
✓ Model the star schema in the Lakehouse — DirectLake doesn't run Power Query at refresh.
✓ Monitor for DirectQuery fallback in Performance Analyzer; size up the Fabric SKU if it happens often.
