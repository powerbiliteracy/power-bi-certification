Configure Dynamic Row-Level Security

Dynamic RLS uses user identity functions and lookup tables to filter data in real-time. Configure RLS without hard-coded values using USERPRINCIPALNAME() and LOOKUPVALUE()—enabling flexible, maintainable security that adapts to organizational changes without model modifications.

Key Principle
Dynamic RLS avoids hard-coded values. Use functions to get current user, then lookup their assigned data access.

Dynamic RLS Basics

⚡ What Is Dynamic RLS?▼

Definition:

✓ User-based filtering
✓ No hard-coded values
✓ Looks up user data

Purpose:

✓ Flexible security
✓ Easy maintenance
✓ Organizational change ready
🔄 Static vs Dynamic▼

Static RLS:

✓ [Region] = "West"
✓ Hard-coded

Dynamic RLS:

✓ [Region] = LOOKUPVALUE(...)
✓ Looks up user's region
✓ Changes without model edit
💪 Benefits of Dynamic▼

Advantages:

✓ No model republishing
✓ Easy user changes
✓ Flexible assignments

Business Impact:

✓ Quick updates
✓ Lower IT burden

Key Functions

👤 USERPRINCIPALNAME()▼

What It Returns:

✓ Current user's UPN
✓ Format: user@domain.com

Example:

✓ john.smith@contoso.com

Use In RLS:

✓ Direct comparison to table
✓ [UserEmail] = USERPRINCIPALNAME()
🔗 LOOKUPVALUE()▼

Purpose:

✓ Look up value in column
✓ Based on criteria

Syntax:

✓ LOOKUPVALUE(result_column, search_column, search_value)

Example:

✓ LOOKUPVALUE(Mapping[Territory], Mapping[UserEmail], USERPRINCIPALNAME())
📋 Combining Functions▼

Pattern:

✓ Get current user
✓ Look up their assignment
✓ Filter data

Complete Example:

✓ [Territory] = LOOKUPVALUE(UserMap[Territory], UserMap[Email], USERPRINCIPALNAME())

Lookup Tables

📊 Creating Lookup Tables▼

Structure:

✓ Column 1: User identifier
✓ Column 2+: Data assignments

Example: UserMap

✓ Email | Territory | Department
✓ john@co.com | West | Sales
✓ mary@co.com | East | Marketing

Updates:

✓ Add new users

Advanced Techniques

Pattern:

Implementation:

✓ Expand lookup result set
