Identify When a Gateway Is Required

Gateway identification is a critical planning step. The decision depends on data location, network accessibility, and connectivity requirements. Master the decision framework to accurately determine when on-premises gateways are necessary for Power BI Service refresh and connectivity.

ScenarioLocationGateway?
SQL Server localOn-PremisesYes

💡
Key Principle
Gateway needed when Power BI Service cannot directly access the data source from the internet due to firewall or network restrictions.

Decision Framework

🔍 Primary Question▼

Ask First:

✓ Can Power BI Service access this data from the internet?

If YES → No gateway needed

If NO → Continue to next question

Is the data on-premises?

✓ Behind corporate firewall?
✓ Local network only?

If YES → Gateway required

📊 Three Categories▼

Category 1: Definitely NO Gateway

✓ Cloud data + internet public
✓ Examples: Azure SQL, Salesforce, Office 365

Category 2: Definitely YES Gateway

✓ On-premises + behind firewall
✓ Examples: Local SQL Server, Oracle DB

Category 3: CHECK First

✓ Hybrid, private endpoints, VPN
✓ Test connectivity before deciding
🎯 Quick Decision Tree▼

Where is the data?

→ Cloud provider? → No gateway
→ Local network? → Likely yes

If local: Behind firewall?

→ Yes → Gateway required
→ No → Check accessibility

If accessible publicly?

→ Yes → No gateway
→ No → Gateway required

Network Assessment

🌐 Network Accessibility▼

Public/Internet-Accessible:

✓ Has public IP address
✓ Publicly routable DNS
✓ No firewall blocking
✓ Result: No gateway needed

Private/Behind Firewall:

✓ Private IP only
✓ Behind corporate firewall
✓ Internal network only
✓ Result: Gateway required
🔐 Special Network Scenarios▼

VPN/Tunnel:

✓ Company VPN to data
✓ Still usually needs gateway

Private Endpoints:

✓ Azure Private Link
✓ May require gateway config

Hybrid Cloud:

✓ Mix of cloud and local
✓ Assess each source individually

Assessment & Planning

📋 Assessment Checklist▼
□ List all data sources
□ Document physical location
□ Check if cloud-hosted
□ Verify internet accessibility
□ Review firewall placement
□ Document network topology
□ Identify refresh requirements
□ Create architecture diagram
🎯 Key Questions to Ask▼
1. Who hosts this data source?
2. Is it a cloud service or self-managed?
3. Can I access it from outside the office?
4. Is there a public endpoint?
5. Does it require VPN?
6. Is it behind company firewall?
7. Does IT require gateway for access?

Real-World Scenarios

Scenario 1: On-Premises SQL Server

Setup:

SQL Server running in data center

Internal network access only

Decision: ✓ Gateway Required

Install enterprise gateway
Configure data source
Scenario 2: Pure Cloud Deployment

Setup:

Azure SQL Database only
No on-premises systems
Cloud-only organization

Decision: ✗ No Gateway Needed

Direct connection from Power BI Service
Configure credentials in Power BI
Scenario 3: Hybrid Environment

Setup:

Azure SQL Database (cloud)
Local Oracle Database
Mix of sources

Decision: Selective

Gateway for Oracle only
Direct for Azure SQL
Scenario 4: Custom Web Service

Setup:

Company API on internal network
No public endpoint
Firewall protected

Decision: ✓ Likely Gateway

Test connectivity first
If needed, install gateway

Best Practices

✓ Planning Strategy▼
□ Document all data sources early
□ Classify by location and accessibility
□ Verify assumptions with IT
□ Test connectivity before design
□ Plan gateway architecture
□ Consider redundancy needs
□ Document decisions
□ Review and update regularly
⚠️ Common Mistakes to Avoid▼
⚠ Assuming all on-premises data needs gateway
⚠ Not checking actual server location
⚠ Ignoring network topology
⚠ Installing gateway without testing
⚠ Wrong gateway type choice
⚠ Not consulting with IT/network team
⚠ Underestimating complexity
