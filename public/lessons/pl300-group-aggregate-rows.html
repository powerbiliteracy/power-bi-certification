What is Grouping and Aggregation?

Grouping combines rows with the same value(s) in specified columns. Aggregation summarizes numerical data (sum, average, count, etc.) for each group. Together, they transform detailed transaction data into summary reports.

Key Benefits

✓ Reduce data volume
✓ Create summary tables
✓ Improve query speed
✓ Simplify reporting
✓ Pre-aggregate data

Real-World Scenarios

Scenario 1: Total Sales by Region
You have 10,000 daily transactions. Group by Region and Sum Amount.

Goal:

Get total sales per region for dashboard.

Approach:

Group By: Region

Aggregate: Sum(Amount)

Table.Group(Source, {"Region"}, {{"Total", each List.Sum([Amount]), type number}})

Result: 4-5 rows (one per region) instead of 10,000

Scenario 2: Average Rating by Product
50,000 customer reviews. Find average rating per product.

Goal:

Create product rating summary.

Approach:

Group By: Product

Aggregate: Average(Rating), Count (for review count)

Table.Group(Source, {"Product"}, {{"AvgRating", each List.Average([Rating]), type number}, {"ReviewCount", each List.Count([Rating]), type number}})

Result: 200 products with ratings (99.6% reduction)

Scenario 3: Monthly Revenue by Region & Product
100,000 daily transactions. Need month/region/product breakdown.

Goal:

Multi-dimensional summary for pivot tables.

Approach:

Group By: YearMonth, Region, Product (3 columns)

Aggregate: Sum(Revenue), Count, Average(Price)

Table.Group(Source, {"YearMonth", "Region", "Product"}, {{"TotalRevenue", each List.Sum([Amount]), type number}, {"TransactionCount", each List.Count([ID]), type number}})

Result: 5,000 rows (95% smaller, enables rich reporting)

Scenario 4: Unique Customer Count by Month
Track customer acquisition across months.

Goal:

See how many unique customers per month.

Approach:

Group By: Month

Aggregate: Count of Distinct CustomerID

Table.Group(Source, {"Month"}, {{"UniqueCustomers", each List.Count(List.Distinct([CustomerID])), type number}})

Result: 12 rows (one per month) with unique customer counts

Best Practices for Grouping & Aggregation

🎯 Aggregate Early in Power Query ▼

Group and aggregate as soon as possible after loading data. Do this before any complex transformations.

Why: Smaller datasets are faster to process. Subsequent transformations work on summary data instead of millions of rows.

Impact: Can reduce query time by 50-80% compared to aggregating in DAX.

📊 Choose the Right Aggregation Function ▼

Sum: Financial totals (revenue, costs)

Average: Ratings, metrics, performance

Count: Transaction count, occurrences

Distinct Count: Unique values (customers, products)

Min/Max: Ranges, outliers

📝 Use Meaningful Column Names ▼

Good: TotalSales, AverageRating, UniqueCustomers

Bad: Sum1, Agg2, Count3

Impact: Clear names make models easier to understand and maintain.

🔍 Handle NULL Values Before Aggregating ▼

NULL values can distort aggregation results.

Options: Remove them, replace with 0, or keep and test impact.

Example: If averaging prices with NULL values, result might be wrong. Remove or replace first.

📌 Order Grouping Columns Strategically ▼

Order grouping columns from highest to lowest level of detail.

Good order: Date → Region → Product → SKU

Bad order: SKU → Product → Region → Date (confusing hierarchy)

Impact: Makes data easier to work with in downstream tables.

💾 Keep Original Key Columns ▼

Always include key columns (IDs, dates, categories) even if not displayed in reports.

Why: Need them to create relationships to other tables.

Example: Keep RegionID even if RegionName is the display column.

🧪 Test Aggregation Results ▼

Verify aggregation is correct by spot-checking results manually.

Steps: Take one group, manually calculate sum/avg/count, compare with grouped result.

Example: If grouping by Region, manually sum one region's sales and verify match.
