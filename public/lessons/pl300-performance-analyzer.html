Identify Poorly Performing Measures, Relationships & Visuals

Performance Analyzer and DAX Query View are diagnostic tools that reveal which measures, relationships, and visuals are slowing down your reports. Identify bottlenecks, measure execution time, optimize DAX queries, and dramatically improve user experience.

IssueDetection MethodResolution
Slow visual loadPerformance Analyzer timingOptimize DAX measure
Inefficient relationshipsDAX Query View executionReview cardinality, add filter
Missing valuesDAX Query resultsFix relationship or data
💡
Key Principle
Measure execution time = Total time - Engine time - Formula engine time. Slow visuals usually stem from inefficient DAX, bad relationships, or cardinality issues. Use Performance Analyzer to identify, DAX Query View to diagnose, then optimize.

Performance Analyzer

🔍 What is Performance Analyzer?▼

Definition:

✓ Built-in Power BI tool for measuring query execution
✓ Shows timing for each visual refresh
✓ Breaks down: Total, Engine, Formula Engine, Other

Purpose:

✓ Identify slow visuals
✓ Measure optimization impact
✓ Benchmark performance
📍 Where is Performance Analyzer?▼

Location:

View → Performance Analyzer

Features:

✓ Start Recording - Captures all visual refreshes
✓ Results show each visual's timing
✓ Click visual name to copy DAX
⏱️ Understanding Timings▼

Time Breakdown:

Total Time: Complete execution (ms)
Engine Time: Data retrieval from model
Formula Engine: DAX calculation
Other: UI rendering, etc.

Example:

• Total: 1,500 ms
• Engine: 1,000 ms (slow query)
• Formula: 400 ms (complex DAX)
• Other: 100 ms
📊 Identifying Slow Visuals▼

Red Flags:

✓ Total time > 1,000 ms (1 second) = slow
✓ Engine time is 90%+ of total = cardinality issue
✓ Formula Engine time high = complex DAX

Action:

✓ Look at query that takes most time
✓ Copy DAX to query view for analysis

DAX Query View

📊 What is DAX Query View?▼

Definition:

✓ IDE for testing DAX queries directly
✓ Shows actual results, timing, row count
✓ Allows query modification and testing

Advantages:

✓ See exactly what data is returned
✓ Test without modifying model
✓ Measure query efficiency
🔗 Relationship Impact▼

Issues Revealed:

✓ Missing values (broken relationship)
✓ Cartesian products (bad cardinality)
✓ Unexpected results (wrong filter)

Example:

DAX returns 1M rows instead of 10K
Indicates many-to-many relationship issue
⚙️ Running Queries▼

Process:

1. Copy DAX from Performance Analyzer
2. Paste into DAX Query View
3. Modify to test changes
4. Execute to see results

Observation:

✓ Duration shows execution time
✓ Results show row count

Common Performance Issues

🔴 Issue 1: Slow DAX Measures▼

Symptoms:

✓ Visual takes 2000+ ms to load
✓ Formula Engine time is high

Common Causes:

• SUMIFS on entire table (no filter)
• CALCULATE with complex conditions
• Nested FILTER with many rows

Solution:

✓ Use more efficient functions (SUMX vs SUMIFS)
✓ Add proper filters to limit rows
✓ Use variables to cache intermediate results
🔴 Issue 2: Cardinality Problems▼

Symptoms:

✓ Engine time dominates (95%+ of total)
✓ Query returns way more rows expected

Causes:

• Many-to-many relationships
• Bi-directional filtering enabled
• Large lookup tables

Solution:

✓ Review relationship cardinality (1:many)
✓ Disable bi-directional if not needed
✓ Use bridge tables for complex relationships
🔴 Issue 3: Inefficient Relationships▼

Symptoms:

✓ Missing data values in visuals
✓ Incorrect totals

Causes:

• Relationship on wrong columns
• Data quality issues in join keys
• Relationship marked as inactive

Solution:

✓ Verify relationship columns match
✓ Check for NULLs in keys
✓ Ensure relationship is active

Solving Performance Problems

✓ Step-by-Step Diagnosis▼

Process:

1. Open Performance Analyzer
2. Start Recording
3. Interact with visuals (click filters, slicers)
4. Review results - identify slowest visual
5. Copy DAX query to clipboard
6. Paste into DAX Query View
7. Analyze query - modify and test

Goal:

✓ Find bottleneck (DAX, relationship, or data)
✓ Implement fix
✓ Re-test with Performance Analyzer
🎯 Optimization Strategies▼

For DAX Measures:

✓ Avoid SUMMARIZE on large tables
✓ Use ADDCOLUMNS instead of nested functions
✓ Cache repeated calculations in variables

For Relationships:

✓ Keep cardinality as 1:many
✓ Disable bi-directional by default
✓ Use bridge tables for complex scenarios

For Visuals:

✓ Reduce fields on axes
✓ Add filters to limit row context
✓ Use aggregated data when possible
📈 Measuring Improvement▼

Before & After:

Measure time before optimization
Apply fix
Re-measure with Performance Analyzer

Example:

• Before: 3,000 ms (slow)
• After: 400 ms (fast)
• Improvement: 87% faster

Real-World Scenarios

Scenario 1: Executive Dashboard - Slow to Load

Problem:

Dashboard takes 30 seconds to load
Users complain about performance

Analysis:

Run Performance Analyzer
Key measure taking 5,000 ms
Copy DAX to query view

Diagnosis:

SUMIFS iterating over entire sales table
No filter pushed to data source

Solution:

Refactor with CALCULATE + FILTER
Result: 400 ms (12x faster)
Scenario 2: Missing Values in Report

Problem:

Some customers showing $0 sales
Expected data exists in source

Analysis:

Use Performance Analyzer to get visual query
Run in DAX Query View
Verify relationship is working

Diagnosis:

Customer-Sales relationship on wrong column
CustomerID vs CustomerCode mismatch

Solution:

Fix relationship to correct column
Data now appears correctly
Scenario 3: Cardinality Issue Identified

Problem:

Query returns 50M rows instead of 100K
Performance tank while running

Analysis:

Performance Analyzer shows 10,000 ms
Engine time is 9,500 ms (95%)
DAX Query View shows massive result set

Diagnosis:

Cartesian product from many-to-many
Sales-Territory relationship misconfigured

Solution:

Create bridge table for relationship
Query now: 500 rows, 50 ms
Scenario 4: Comparing Query Performance

Problem:

Two ways to write same measure
Don't know which is faster

Solution:

Test both in DAX Query View
Version A: 2,000 ms, returns 100K rows
Version B: 200 ms, returns 100K rows

Result:

Version B is 10x faster
Implement optimized version
