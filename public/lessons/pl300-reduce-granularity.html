Reduce Granularity for Performance

Reducing granularity means loading data at a coarser level: daily instead of hourly, monthly instead of daily. Dramatically reduces rows, file size, and memory usage.

LevelRows (1M trans)File SizeRefresh Time
Hourly8.76M rows800 MB90 min
Daily365K rows40 MB5 min
Monthly12K rows2 MB30 sec
💡
Key Principle
Granularity = Data detail level. Reducing from daily to monthly: 96% fewer rows, 95% smaller file, 10x faster refresh. Perfect for dashboards, summaries, historical analysis where hourly/daily granularity unnecessary.

Understanding Granularity

🔍 What is Granularity?▼

Definition:

✓ Level of detail in data
✓ How fine the time/dimension slice

Examples:

• Hourly: 24 records per day per store
• Daily: 1 record per day per store
• Monthly: 1 record per month per store
• Yearly: 1 record per year per store
📊 Granularity Levels (Time)▼

Finest to Coarsest:

1. Second-level (1M+ rows/day) - Very slow
2. Minute-level (60K rows/day) - Slow
3. Hourly (1K+ rows/day) - Medium
4. Daily (1 row/day) - Fast
5. Weekly (1 row/week) - Very fast
6. Monthly (1 row/month) - Extremely fast
7. Yearly (1 row/year) - Instant
🎯 When to Reduce Granularity▼

Good Candidates:

✓ Historical data (2020, 2021)
✓ Trend analysis (quarterly views)
✓ Executive dashboards (monthly)
✓ Archived periods (before current year)

Bad Candidates:

⚠ Current period (need daily detail)
⚠ Transactional analysis (order detail)
⚠ Real-time monitoring (second-level)

Strategies to Reduce Granularity

📅 Strategy 1: Time-Based Aggregation▼

Concept:

Load current month/year in detail
Load history in coarser granularity

Example:

Current year: Daily (2024, detail needed)
Prior years: Monthly (2020-2023, archive)

Impact:

✓ 2024: 365 daily records
✓ 2023-2020: ~48 monthly records
✓ Total: ~413 vs 1,460 (71% smaller)
📊 Strategy 2: Dimension Aggregation▼

Concept:

Load all SKUs for current period
Load category-level for historical

Example:

Current: All 50K SKUs
History: 20 Categories only

Benefits:

✓ Current detail for deep analysis
✓ History fast & responsive
🔄 Strategy 3: Pre-Aggregation at Source▼

Best Practice:

✓ Aggregate in SQL/source (not Power BI)
✓ Load pre-summarized data
✓ Much faster than in-memory aggregation

Benefits:

✓ Less data transferred
✓ Faster refresh
✓ Source can optimize
🎯 Strategy 4: Separate Tables by Granularity▼

Approach:

Table 1: FactSalesDaily (current year)
Table 2: FactSalesMonthly (historical)
Union in Power Query or Power BI

Advantages:

✓ Can manage separately
✓ Different refresh schedules
✓ Different dimensions per table

Real Examples

📈 Example 1: Sales Trend (Daily vs Monthly)▼

Current Year (2024): Daily

Jan 1: $5,000, Jan 2: $5,100, ... (365 records)

Prior Years (2020-2023): Monthly

2023-01: $155,000 (sum of Jan 2023)
2023-02: $148,000 (sum of Feb 2023) ...

Benefit:

✓ 413 records vs 1,460 (71% smaller)
📦 Example 2: Inventory (SKU vs Category)▼

Current: All SKUs

Electronics: 1000 SKUs, Clothing: 5000 SKUs

Historical: Category Only

Electronics: $50M (all SKUs combined)
Clothing: $30M

Result:

✓ Current: 6K SKU records
✓ History: 20 category records
⏰ Example 3: Web Analytics (Hourly to Daily)▼

Original Data (Hourly):

365 days × 24 hours = 8,760 records/year
10 years = 87,600 records (slow!)

Optimized (Daily):

365 records/year × 10 = 3,650 records (fast!)

Trade-off:

✓ Can't see hourly trends in 2015
✓ But can see daily/monthly trends
✓ 96% fewer rows

Real-World Scenarios

Scenario 1: Financial Reporting (Monthly History)
Detailed current month, monthly archives

Current Month (Detail):

Daily transactions (Jan 1-31)

Prior Months (Archive):

Monthly roll-up (Dec, Nov, Oct...)

Result:

✓ Current detail for deep analysis
✓ Historical summary for trends
✓ ~100 rows vs 3,650 rows
Scenario 2: Executive Dashboard (Quarterly)
High-level, performance-focused

Data Load:

Only last 4 quarters (Q4 2023 - Q3 2024)
Aggregated to quarterly summaries

Benefits:

✓ Instant load (tiny dataset)
✓ Responsive dashboards
Scenario 3: Multi-Year Analytics (Yearly)
Long-term trends at yearly level

Data:

10 years of data (2015-2024)
Aggregated to yearly level
Only 10 rows total

Use Cases:

✓ Year-over-year comparisons
✓ Long-term trends
Scenario 4: Hybrid Approach (Mixed)
Best of both: detail + performance

Structure:

Table 1 (Daily): Current year
Table 2 (Monthly): Prior 3 years
Table 3 (Yearly): Historical (5+ years)

Result:

✓ 406 rows total
✓ Current detail available
✓ History fast & responsive

Best Practices

✓ Granularity Best Practices▼
✓ Ask: "Do users need hourly detail for 2020?"
✓ Load current period in detail
✓ Load historical in summaries
✓ Pre-aggregate at source (SQL, not Power BI)
✓ Document granularity levels per table
✓ Consider hybrid approach
✓ Test performance before/after
⚠️ Common Mistakes▼
⚠ Loading 20 years of hourly data
⚠ Aggregating in Power BI instead of source
⚠ Not considering user needs
⚠ Losing important dimensions
⚠ Not documenting what's aggregated
