Reference vs Duplicate Queries

In Power Query, you can build on existing queries in two ways: reference (use another query's output) or duplicate (independent copy). Each has different implications for performance, maintainability, and resource usage.

AspectReferenceDuplicate
DependencyDepends on sourceIndependent
Code SizeMinimal duplicationFull code copied
Refresh SpeedFaster (shared load)Slower (separate loads)
File SizeSmallerLarger
MaintenanceUpdate once = all updateUpdate each separately
💡
Key Principle
Reference queries are better for DRY (Don't Repeat Yourself) and shared data. Duplicate queries are better for independence and avoiding complex dependencies. Choose based on your data flow and model complexity.

Performance & Maintenance Impact

⚡ Refresh Performance ▼

Reference Query Impact:

• Source loads once, referenced queries reuse output
• Query folding applies up the dependency chain
• Faster overall refresh (no duplicate processing)

Duplicate Query Impact:

• Each duplicate loads source independently
• Same transformations execute multiple times
• Slower refresh (duplicated work)
Example: 10 queries from same source
Reference: Source loads 1x, all 10 use cached output
Duplicate: Source loads 10x independently
📁 File Size & Memory ▼

Reference Query Impact:

• Minimal M code duplication
• Smaller .pbix file
• Less memory during refresh

Duplicate Query Impact:

• M code copied for each duplicate
• Larger .pbix file (code bloat)
• More memory for transformation steps
Example: 5 duplicates of 50-step query
Reference: 50 steps stored once
Duplicate: 50 steps × 5 = 250 steps stored
🔧 Maintenance & Changes ▼

Reference Query Impact:

• Change source once = updates all dependents
• Single source of truth
• Easier to maintain consistency
• Risk: One broken source breaks everything

Duplicate Query Impact:

• Must update each duplicate separately
• Easy to get out of sync
• More maintenance burden
• Better isolation if intentionally different
Example: Fix date format in 5 queries
Reference: Edit once, all 5 update automatically
Duplicate: Edit 5 times manually, easy to miss one

Real-World Scenarios

Scenario 1: Clean Data Once, Use Many Times
One raw data source, multiple cleaned versions for different uses

Situation:

Source CSV with messy data
Need: Sales analysis, Customer analysis, Inventory analysis

Best Approach: REFERENCE

RawData (Source) ↓ CleanedData (Reference - cleaning logic) ├→ SalesData (Reference) ├→ CustomerData (Reference) └→ InventoryData (Reference)

Why Reference?

✓ Clean once at CleanedData level
✓ All dependent queries inherit cleaning
✓ Change format rule = updates all 3
✓ Minimal file size (no code duplication)
Scenario 2: Different Transformations Needed
Same source, completely different processing for each

Situation:

Sales data needs: Aggregation by day/product
Customer data needs: Aggregation by region/segment
Only common: Load and clean source

Best Approach: REFERENCE then consider DUPLICATE

Reference up to common point, then duplicate if truly independent logic.

Best Practices & Decision Framework

📋 Decision Framework ▼

Ask these questions in order:

1
Same data source?
Uses output from existing query?
YES → Consider Reference
2
Same transformations?
Identical cleaning steps needed?
YES → Reference | NO → Consider Duplicate
3
Dependency chain depth?
Would reference create 5+ levels?
YES → Break chain with Duplicate
4
Truly independent logic?
Will evolve separately from source?
YES → Duplicate | NO → Reference
✓ When to Use Reference ▼

✓ Use Reference:

Shared data transformation
Multiple queries use same cleaning
Want single source of truth
Performance matters
File size is concern
Maintenance is priority
✓ When to Use Duplicate ▼

⚠ Use Duplicate:

Completely different logic
Avoid complex dependencies
Breaking chain reduces risk
Query evolves independently
Small queries (not critical)
Intentionally separate concerns
