Implement Row-Level Security Roles

Row-level security (RLS) filters data based on user identity. Create roles with DAX rules to restrict which rows users can see. Enable fine-grained data access control—users with same dataset access see different data based on their assigned role, protecting sensitive information while maximizing self-service analytics.

Security LayerControl LevelGranularity
Workspace roleItem accessBroad
Item accessReport/dashboardFine
RLS roleIndividual rowsFinest
💡
Key Principle
RLS filters data at the row level. Users see only data relevant to their role/identity. Works automatically in all reports using that dataset.

DAX Rules for RLS

📝 Writing DAX Rules▼

Basic Syntax:

✓ [Column] = [Function]
✓ Example: [Region] = "West"

Common Functions:

✓ USERNAME()
✓ USERPRINCIPALNAME()
✓ LOOKUPVALUE()
👤 Using USERNAME()▼

Function:

✓ Returns current user
✓ Format: DOMAIN\Username

Example:

✓ [Manager] = USERNAME()
✓ Show only rows for logged-in user
🔗 Using LOOKUPVALUE()▼

Purpose:

✓ Look up user's territory
✓ From mapping table

Example:

✓ [Territory] = LOOKUPVALUE(Mapping[Territory], Mapping[User], USERNAME())
✓ Flexible user-to-data mapping
⚠️ Complex Rules▼

Possibilities:

✓ OR conditions
✓ AND conditions
✓ Multiple tables

Example:

✓ ([Region] = "North") || (USERNAME() = "admin")
✓ Complex business logic

Real-World Scenarios

Scenario 1: Sales Territory

Requirement:

Sales managers see own territory
Different regions, same report
Automatic filtering by user

Solution:

Create Sales Manager role
DAX: [Territory] = LOOKUPVALUE(TerritoryMap[Territory], TerritoryMap[Manager], USERNAME())
Each manager sees only their data
Scenario 2: Department Access

Requirement:

HR employees see HR data only
Finance sees finance data
No cross-access

Solution:

Create dept-based roles
DAX: [Department] = "HR"
Apply to all sensitive tables
Scenario 3: Multi-Tenant SaaS

Requirement:

Multiple customers using app
Completely isolated data
Shared dataset/reports

Solution:

Customer role
DAX looks up customer from user
Filter all tables by customer
Scenario 4: Management Hierarchy

Requirement:

Managers see team data
Directors see multiple teams
Hierarchy-based access

Solution:

Mapping table with hierarchy
RLS evaluates role in hierarchy
Different users, different views
