Configure Access to Semantic Models

Semantic Model Access Basics

💾 What Is Semantic Model Access?▼

Definition:

✓ Dataset-level permissions
✓ Control who can use data
✓ Enable self-service

Purpose:

✓ Protect sensitive data
✓ Allow analytics
✓ Maintain governance
🎯 Why Separate from Reports?▼

Key Difference:

✓ Dataset access = permission to use data
✓ Report access = permission to view report

Both Required:

✓ To see report: need report access
✓ To build on dataset: need dataset access
🔐 Who Manages Dataset Access?▼

Can Configure:

✓ Dataset owner
✓ Workspace admin
✓ Member with access

Considerations:

✓ Data governance team
✓ Data steward role

Semantic Model Permissions

👁️ Read Permission▼

What It Allows:

✓ View dataset metadata
✓ See tables and columns
✓ Understand structure

Cannot Do:

✗ Create reports
✗ Query data
✗ Build analyses

Use Case:

✓ Documentation access
✓ Data discovery
🔨 Build Permission▼

What It Allows:

✓ Create reports/dashboards
✓ Query all data
✓ Build analyses
✓ Create visuals

Cannot Do:

✗ Modify dataset
✗ Change structure
✗ Manage permissions

Use Case:

✓ Self-service analytics
✓ Report creation
🔄 Reshare Permission▼

What It Allows:

✓ Share with others
✓ Grant build access
✓ Distribute access

Typically Combined With:

✓ Build permission
✓ For collaborative teams

Use Case:

✓ Team leads sharing
✓ Collaborative projects
👑 Admin Permission▼

What It Allows:

✓ Full control
✓ Manage permissions
✓ Change dataset settings
✓ Configure endorsement

Use Case:

✓ Dataset owner
✓ Data stewards
✓ Governance team

Granting Access to Semantic Models

👥 How to Grant Access▼

Steps:

1. Dataset settings
2. Click "Grant access"
3. Search users/groups
4. Select permission level
5. Add

Location:

✓ Workspace → Dataset → Settings
✓ "Grant access" button
👤 Individual vs Group Access▼

Individual Users:

✓ Search by email
✓ One-by-one grant
✓ Good for exceptions

Security Groups:

✓ Team-based access
✓ Easier to manage
✓ Bulk grant
✓ Dynamic membership
📋 Permission Selection▼

Choose Based On:

✓ What user needs to do
✓ Principle of least privilege
✓ Role and responsibilities

Combinations:

✓ Build alone = create reports
✓ Build + Reshare = share access
✓ Read only = awareness

Managing Semantic Model Access

📊 Viewing Current Access▼

Check Who Has Access:

✓ Dataset settings
✓ Scroll to access section
✓ See all users/groups

Information Shown:

✓ Name/email
✓ Permission level
✓ Action buttons
🔄 Modifying Access▼

Change Permissions:

✓ Click user in access list
✓ Select new permission level
✓ Save changes

Effective Immediately:

✓ User permissions update
✓ No delay
❌ Removing Access▼

Process:

1. Access management screen
2. Find user/group
3. Click remove
4. Confirm

Result:

✓ Immediate removal
✓ Can no longer build
✓ Existing reports remain

Advanced Access Control

🔐 RLS with Dataset Access▼

Important Combination:

✓ Dataset access = can build
✓ RLS = limits data seen
✓ Both apply together

Example:

✓ User has build access
✓ RLS restricts to their region
✓ Can only build with region data
📈 Object-Level Security▼

Advanced Feature:

✓ Restrict specific columns/tables
✓ Premium only feature
✓ Hide sensitive columns

Use Case:

✓ Hide salary data
✓ Restrict SSN access
🎯 Access Inheritance▼

Workspace Role Impact:

✓ Admins can build automatically
✓ Members can build (if granted)
✓ Contributors need explicit grant
✓ Viewers need explicit grant

Note:

✓ Dataset access separate from workspace

Real-World Scenarios

Scenario 1: Self-Service Analytics

Requirement:

Sales analysts create reports
On shared sales dataset
Independent of IT

Solution:

Grant build access to analyst group
To shared dataset
Include reshare option
Scenario 2: Sensitive Data Protection

Requirement:

HR dataset very confidential
Only HR team access
No self-service

Solution:

No build access granted
Only IT/data team admin
Reports shared separately
Scenario 3: Read-Only Discovery

Requirement:

Executives want data awareness
See what data exists
Cannot create reports

Solution:

Grant read-only permission
See metadata and structure
Understand available data
Scenario 4: Collaborative Teams

Requirement:

Finance team owns dataset
Share with accounting team
Both can create reports

Solution:

Build + reshare to accounting
Team can create independently
Or grant to both teams

Best Practices

✓ Access Strategy▼
□ Map data sensitivity levels
□ Define access tiers
□ Use groups for teams
□ Document data ownership
□ Review access regularly
□ Audit for compliance
□ Minimize admin users
□ Combine with RLS when needed
⚠️ Common Mistakes▼
⚠ Too open - Restrict carefully
⚠ Too restrictive - Enable analytics
⚠ Individual grants - Use groups
⚠ No documentation - Document
⚠ Forgetting RLS - Combine both
⚠ No review - Audit regularly
⚠ Too many admins - Limit admin
